## Findings

### Finding 1: UNIQUE INDEX migration can brick the app on deploy if duplicate rows already exist
- **Severity:** Critical
- **Location:** Phase 2, section "Architecture" step 1 (`SCHEMA_SQL` UNIQUE INDEX)
- **Flaw:** `init_schema()` runs `CREATE UNIQUE INDEX IF NOT EXISTS ux_appointments_slot ON appointments (date, time) WHERE status='confirmed'` unconditionally and is invoked at the top of every `list_appointments`, `get_appointment`, and `add_appointment` call (storage.py:132-177). If the production DB already has ≥2 `confirmed` rows sharing `(date,time)` — which is exactly the bug C2 is fixing, so pre-existing duplicates are plausible in a live system that has been running with this race — `CREATE UNIQUE INDEX` fails with a Postgres error. `_schema_ready` is only set `True` after a successful `cur.execute(SCHEMA_SQL); conn.commit()` (storage.py:91-99), so the failure is not cached-away; it repeats on the very next request and every request after that.
- **Failure scenario:** Deploy this fix to production. Any historical double-booking (the exact scenario C2 exists to prevent) turns the fix into a full outage: every route touching `storage.*` (booking, admin, worker) starts raising an unhandled DB exception instead of just failing the one duplicate insert.
- **Evidence:** storage.py:91-99 (`init_schema`, no try/except), storage.py:132-177 (all callers invoke `init_schema()` first); phase-02 plan has no step for detecting/cleaning existing duplicate `(date,time)` confirmed rows before adding the constraint.
- **Suggested fix:** Add a pre-migration dedup step (e.g. a one-off script or manual query to demote all-but-one duplicate confirmed row to `cancelled` before running `init_schema()` in prod), or wrap the `CREATE UNIQUE INDEX` statement in its own try/except inside `init_schema()` so a migration failure doesn't take down unrelated read paths.

### Finding 2: "1 process" is treated as equivalent to "no concurrency" for both C2 and C3, which is not verified anywhere in the codebase
- **Severity:** Critical
- **Location:** Phase 3, "Architecture" (OrderedDict without a lock); plan.md "Quyết định đã chốt với user"
- **Flaw:** The plan's justification for skipping any locking mechanism is "production runs a single Flask process." No Procfile, gunicorn config, or WSGI entrypoint exists in the repo to confirm the threading model of that single process (only `app.run(debug=True, host="0.0.0.0", port=5001)` at app.py's `__main__`, which is dev-only). A single OS process can still serve many requests concurrently (gunicorn `--threads`, gevent workers, or Flask's own dev server with `threaded=True`), and ISSUES.md **already lists this exact gap as a separate, deferred Medium item**: "Sửa session dict không có khóa (lock) khi Flask threaded → state có thể bị ghi đè" (ISSUES.md:67).
- **Failure scenario:** Two concurrent requests for different sessions both trigger the eviction path (`SESSIONS.popitem(last=False)`) or `move_to_end()` around the same dict mutation window. Unlike the current plain `dict`, which tolerates concurrent `SESSIONS[sid][...] = x` writes without raising, `OrderedDict.popitem`/`move_to_end` interleaved with insertion under real thread concurrency can corrupt ordering or raise (e.g., `KeyError` from `popitem` racing against a state where the dict was just mutated by another thread). The "fix" for a DoS/reliability issue introduces a new reliability edge case that is more failure-prone under concurrency than the code it replaces, and the plan does not add a lock nor confirm single-threaded deployment before shipping.
- **Evidence:** ISSUES.md:67 (existing deferred finding), phase-03.md:33-56 (OrderedDict design, no `threading.Lock`), no `Procfile`/`gunicorn.conf.py`/`wsgi.py` found in repo root.
- **Suggested fix:** Either (a) confirm and document the exact prod WSGI command (workers=1, threads=1) before accepting the no-lock design, or (b) wrap `SESSIONS` mutations in a `threading.Lock` regardless — cheap insurance that removes the unverified assumption entirely.

### Finding 3: Rollback/connection-state after `UniqueViolation` is explicitly left unresolved
- **Severity:** High
- **Location:** Phase 2, "Related Code Files" ("có thể cần rollback connection khi lỗi") vs. "Architecture" step 2/3
- **Flaw:** The plan itself flags this as an open question ("có thể cần...") but never resolves it in the Architecture or Implementation Steps. `add_appointment()`'s `with _connect() as conn, conn.cursor() as cur:` block will raise `UniqueViolation` mid-`with`; whether psycopg3's context manager auto-rolls-back that specific connection before `book_appointment`'s except-block opens a *new* connection via `_confirmed_at()` is asserted, not verified or tested. Given the storage docstring recommends "Connection pooler / Transaction" mode (storage.py:12), pooler-level transaction semantics can behave differently than a direct connection under aborted-transaction states.
- **Failure scenario:** If the aborted connection/transaction is somehow reused or the pooler serves a connection still in an aborted-transaction state to the very next `_confirmed_at()` read, that read could itself fail (`InFailedSqlTransaction`), causing `book_appointment`'s error-recovery path to throw a *second*, unhandled exception instead of returning the intended "slot taken" response.
- **Evidence:** phase-02.md:53 ("có thể cần rollback connection khi lỗi") — an acknowledged but unresolved risk in a Critical/P1 fix; no test in phase-02's TDD plan actually connects to a real Postgres transaction-pooled connection to verify this (phase-02.md:60-75 confirms all tests are monkeypatch-based, explicitly *not* against real Postgres).
- **Suggested fix:** Require at least one real-Postgres integration test (or explicit sign-off that this was manually verified against the actual Supabase pooler config) before marking C2 done, not just the monkeypatch unit test.

### Finding 4: C1's normalization fix assumes DB-managed patterns are already clean; life-safety patterns are exactly the ones an admin edits by hand
- **Severity:** High
- **Location:** Phase 1, "Architecture" / Implementation step 2
- **Flaw:** The proposed fix does `_strip_accents(p)` on each pattern `p` but never runs `_normalize(p)` on it (no lowercasing, no whitespace/punctuation collapsing) — it only normalizes the *input text*. This exactly mirrors the same latent gap already present in `triage.py:63-66` (`_contains_word(norm, kw)` where `kw` is never run through `_normalize`). For the hardcoded seed list this happens to work because all `_SEED_EMERGENCY_PATTERNS` entries are already lowercase, punctuation-free (safety.py:21-33). But safety.py:76-80 explicitly documents that these patterns are **also loaded live from a Supabase table `safety_patterns` managed via dashboard** ("Quản lý online tại Supabase"), with no schema-level constraint forcing lowercase/normalized text.
- **Failure scenario:** An on-call staff member adds a new emergency pattern via the Supabase dashboard with natural capitalization/punctuation (e.g. "Đau ngực, dữ dội!" or "Khó thở Nặng"), which is a realistic real-world edit path since this table is explicitly designed for non-developer, live editing. Because the pattern isn't lowercased/punctuation-stripped before `_strip_accents`, `_contains_word` will fail to match even a byte-identical no-accent user message, silently defeating the exact life-safety goal C1 exists for — for the one data source (DB) that's most likely to receive imprecise, hand-typed input.
- **Evidence:** safety.py:76-80 (DB-managed patterns), triage.py:63-66 (unnormalized `kw`), phase-01.md:56-64 (fix code only calls `_strip_accents(p)`, never `_normalize(p)`).
- **Suggested fix:** Apply `_strip_accents(_normalize(p))` to every pattern, matching exactly what's already done for the input text, regardless of whether patterns come from seed or DB.

### Finding 5: C4's "ownership" check is authorization-by-unauthenticated-bearer-token with no validation of session-id strength, and the plan's own summary contradicts the phase spec
- **Severity:** High
- **Location:** Phase 4, "Architecture" step 2; plan.md line 66 (File Ownership section) vs phase-04.md
- **Flaw:** `resolve_sid()` (app.py:31-38) accepts a fully client-chosen `session` string from the JSON body with zero format/length/entropy validation, and unconditionally writes it into the Flask session cookie (`session["sid"] = sid`) as a side effect of being called — including from the newly-added call inside `download_ics`. The "ownership" check added in Phase 4 (`appt.get("session") != sid`) is therefore only as strong as the assumption that `session` values are always high-entropy and never guessable/reused — an assumption ISSUES.md itself lists as a residual **Medium** risk ("Client tự chọn `session` id (giảm nhẹ nhờ entropy uuid4)", ISSUES.md:64), not a solved problem. Phase 4's own Risk Assessment discusses code entropy (`_generate_code`) and mobile-app breakage at length but never discusses the strength/validation of the `session` identifier it now uses as the sole authorization credential for sensitive health data.
- **Failure scenario:** Nothing in `resolve_sid` stops a client from setting `"session": "1"` (or any other low-entropy value) when booking, and nothing stops a different client from later probing `/api/ics/<code>` with the same guessed low-entropy `session` values to hit other users who also picked predictable ids (e.g., testers, scripted clients, or any future client implementation that doesn't generate a proper UUID). The plan treats "matches session field" as equivalent to "is the owner," without adding any check that the `session` value looks like a real server/app-issued random id.
- **Evidence:** app.py:31-38 (`resolve_sid`, no validation), ISSUES.md:64 (Medium: client-chosen session id, entropy-dependent), phase-04.md Risk Assessment (lines 106-115) — no mention of session-id validation.
- **Also (fact-check FAILED):** plan.md:66 states Phase 4 adds "kiểm tra quyền sở hữu trong `book_appointment`/route" but phase-04.md's actual Architecture (lines 38-68) only modifies `app.py::download_ics`; `booking.book_appointment` is not touched for ownership at all. The top-level plan overstates what the phase spec actually implements.
- **Suggested fix:** At minimum, validate that `session` matches an expected format (e.g., 32-hex uuid4) server-side before accepting it as an authorization credential, and correct the plan.md summary to match phase-04's actual scope.

### Finding 6: Phase 5's blanket `except Exception` around the whole per-appointment body will also swallow bugs inside `_send_for`'s side-effecting calls, masking the already-known H2 issue further
- **Severity:** Medium
- **Location:** Phase 5, "Architecture" (single `try/except Exception` wrapping both the date-parse and the `_send_for` call)
- **Flaw:** The try/except wraps both (a) malformed-data parsing (the actual C5 target) and (b) `_send_for(appt, rule)`, which itself calls `push.send_push` and `booking.mark_reminder_sent`. If `_send_for` partially succeeds (push sent) but `mark_reminder_sent` throws (e.g. transient DB error), this new catch-all logs a generic `[SKIP] ... lỗi xử lý nhắc lịch: {e}` and moves on — the reminder was actually sent but not recorded as sent, so it will be re-sent on the next scan (duplicate notification) or, depending on where in the loop the exception hit, the dedup bookkeeping silently diverges from reality. The plan's own Risk Assessment acknowledges the general risk ("có thể che giấu bug thật") but doesn't distinguish this specific data-corruption-adjacent case from a harmless malformed-record skip.
- **Failure scenario:** Transient Postgres/network blip during `mark_reminder_sent` right after a successful `push.send_push` → patient gets duplicate reminder pushes on the next `--watch` cycle, and the log line gives no signal that this differs from a genuinely broken record.
- **Evidence:** phase-05.md:34-58 (single try/except spanning both date-parse and `_send_for`), phase-05.md:99-105 (Risk Assessment acknowledges but doesn't scope the distinction).
- **Suggested fix:** Keep the outer catch-all for record-level parsing errors (C5's actual scope), but note in the log line whether the failure occurred before or after `push.send_push` succeeded (e.g. include exception type/`repr(e)` and which step failed) so H2 remains debuggable rather than further obscured by this new blanket handler.
