# Red Team Review — Fix Critical Issues (C1-C5) Plan
Role: Assumption Destroyer / Scope Auditor

## Finding 1: New UNIQUE index can fail at schema-init time if duplicate confirmed slots already exist in production
- **Severity:** Critical
- **Location:** Phase 2, section "Architecture" step 1 + Success Criteria
- **Flaw:** The plan adds `CREATE UNIQUE INDEX IF NOT EXISTS ux_appointments_slot ON appointments (date, time) WHERE status = 'confirmed'` to `SCHEMA_SQL`, executed inside `storage.init_schema()`. `IF NOT EXISTS` only guards against re-running the DDL — it does NOT tolerate pre-existing duplicate rows that violate the new uniqueness constraint. `storage.init_schema()` (storage.py:91-99) has no try/except around `cur.execute(SCHEMA_SQL)`, and `init_schema()` is invoked lazily from `list_appointments`, `get_appointment`, `add_appointment` etc. on the very first request after deploy (storage.py:133-135, 143-144, 158-159).
- **Failure scenario:** C2 exists precisely because double-booking has been possible in production. If even one pair of `(date, time)` confirmed appointments already collided before this fix ships (plausible, since that's the bug being fixed), the first request touching storage after deploy raises `psycopg.errors.UniqueViolation` uncaught inside `init_schema()`, `_schema_ready` stays `False`, and **every subsequent request that touches storage fails the same way** — the entire app becomes unusable in production until someone manually finds and deletes/reassigns the offending duplicate row(s) from the DB and re-deploys.
- **Evidence:** `storage.py:91-99` (`init_schema`, no try/except); phase-02 Architecture step 1 (plan text lines 35-40) never mentions checking for or cleaning existing duplicates before adding the constraint.
- **Suggested fix:** Before/at deploy, add a one-off cleanup step (e.g. a script or migration note) that detects and resolves existing `(date,time)` duplicates among `status='confirmed'` rows, or wrap `init_schema()`'s DDL execution so failures don't permanently corrupt `_schema_ready` and don't take down every route — at minimum, document the pre-flight check as a required manual step in the phase's Implementation Steps, not left implicit.

## Finding 2: Accent-insensitive fix stays case-sensitive for DB-sourced patterns, silently defeating the guardrail it's meant to strengthen
- **Severity:** Critical
- **Location:** Phase 1, section "Implementation Steps (TDD)" step 2 (Green) and "Risk Assessment"
- **Flaw:** The proposed code normalizes only the user's `text` fully (`_strip_accents(_normalize(text))`, lowercases + strips punctuation) but normalizes each pattern `p` with `_strip_accents(p)` alone — never running `p` through `_normalize()`. `_contains_word` (triage.py:40-45) is a literal, case-sensitive substring check (`f" {needle} " in f" {haystack} "`). `EMERGENCY_PATTERNS`/`DIAGNOSIS_REQUEST_PATTERNS`/`HANDOFF_PATTERNS` are not fixed hardcoded lowercase constants — they are loaded from Supabase via `storage.list_safety_patterns()` (storage.py:301-313) with zero normalization applied at the DB layer, and are admin-editable. The plan's Risk Assessment (line 82) claims this "đã là hành vi của `triage.py`, kế thừa đúng" — but triage.py's equivalent (`_DENTAL_PARTS`, `_mention_tokens`) only ever applies this pattern to hardcoded, already-lowercase Python constants, not to freeform DB input. That precondition does not hold for `safety.py`.
- **Failure scenario:** An admin (non-technical, editing the `safety_patterns` table in the Supabase dashboard) enters "Đau ngực dữ dội" or "Khó Thở" with any capitalization or leading/trailing whitespace. `_strip_accents(p)` preserves the case (e.g. "Dau nguc du doi"), so it never matches the always-lowercase `norm_na` derived from user text — the emergency pattern is silently inert for that entry, even though it appears correctly in the DB and even though the whole point of this phase is to make emergency detection more robust. This is a life-safety detection path (calls for 115).
- **Evidence:** phase-01 lines 60-63 (proposed `check_emergency` code); triage.py:40-45 (`_contains_word`, case-sensitive `in`); storage.py:301-313 (`list_safety_patterns`, raw passthrough, no `.lower()`/normalization).
- **Suggested fix:** Apply the same `_strip_accents(_normalize(p))` treatment to patterns as to text, for both seed and DB-sourced entries — either at load time in `safety._load_patterns()` or inline in each matching function.

## Finding 3: `reset_session()` desyncs OrderedDict eviction order from `_last_seen` recency, defeating "evict least-recently-used"
- **Severity:** High
- **Location:** Phase 3, section "Architecture" (`get_session` steps) — `reset_session()` is not covered
- **Flaw:** The architecture describes cap/TTL eviction purely in terms of `get_session()`: `move_to_end()` on hit, `popitem(last=False)` when over cap. But `reset_session(session_id)` (called from `chatbot.py:63` and `chatbot.py:84`, both times with a session_id that is virtually always already present in `SESSIONS`) does `SESSIONS[session_id] = _new_session()` directly. Reassigning an existing key's value in an `OrderedDict` does **not** move it to the end — only explicit `move_to_end()` or delete+reinsert does. The new value does get a fresh `_last_seen`, but the dict's insertion-order position (which is what `popitem(last=False)` actually uses for cap eviction) stays wherever the session originally was.
- **Failure scenario:** A session that is mid-conversation gets reset (e.g. via the cancel/duplicate-booking flow at chatbot.py:63,84) early in its lifetime, so it sits near the front of `SESSIONS`'s iteration order. Under load, once `_MAX_SESSIONS` is reached, `popitem(last=False)` evicts this session first — even though its `_last_seen` is now the most recent of all sessions — silently dropping an actively-in-progress conversation ahead of genuinely idle ones. This contradicts the plan's own requirement: "Session cũ nhất (theo lần truy cập gần nhất...) bị loại bỏ khi vượt trần" (phase-03 line 27).
- **Evidence:** chatbot.py:41-42 (`reset_session`), chatbot.py:63, chatbot.py:84 (call sites, both post-`get_session`, i.e. key already exists); phase-03 lines 43-50 (Architecture only specifies `get_session` behavior, no mention of `reset_session` needing `move_to_end()`); the test plan (phase-03 lines 65-76) has no case exercising `reset_session()` interaction with eviction order.
- **Suggested fix:** Either route `reset_session()` through the same insert/refresh logic as `get_session()` (call `move_to_end()` after reassignment), or explicitly document/test that reset sessions keep their original position and confirm that's acceptable.

## Finding 4: "1 Flask process" assumption underpinning the whole C3 design is unverifiable — no process/worker config exists in the repo
- **Severity:** Medium
- **Location:** plan.md lines 23-25 ("Quyết định đã chốt với user"); Phase 3 Overview lines 16-19
- **Flaw:** The entire C3 design (in-memory `OrderedDict` cap, no Redis) is justified by "Production chạy 1 Flask process duy nhất." This is stated as a settled fact, but grep across the repo root finds no `Procfile`, no gunicorn config, no `wsgi.py`, no `WEB_CONCURRENCY`/worker-count env var, and no deployment manifest of any kind (`find` for Procfile/render.yaml/wsgi/gunicorn returned nothing). The only run instructions in `README.md`/`setup.sh` are `python app.py` (dev) and `reminder_worker.py --watch`. Nothing in the codebase enforces or even documents the single-process constraint this fix's correctness depends on.
- **Failure scenario:** If production is later deployed behind gunicorn/uWSGI with multiple workers (a very common default for Flask in production, e.g. `gunicorn -w 4 app:app`), each worker process gets its own independent `SESSIONS` OrderedDict with its own 2000-entry cap — the assumption silently breaks with no error, no warning, and no test catching it (all Phase 3 tests run in-process). This doesn't cause unbounded OOM (still capped per-worker) but does 4x the memory ceiling with no capacity planning input, and it doesn't fix the pre-existing "same user, different worker → state reset" bug that already exists today for `SESSIONS`-based routing.
- **Evidence:** `find . -maxdepth 1 -iname "Procfile*" ...` → no results; `requirements.txt` has no `gunicorn`; plan.md:23-25 states the decision as given fact with no enforcement mechanism proposed.
- **Suggested fix:** Either add an explicit runtime guard/log line asserting single-worker operation (e.g. fail fast or warn if `WEB_CONCURRENCY`/gunicorn worker count > 1 is detected), or document the constraint prominently in `README.md`/deployment docs so it isn't silently violated by a future deploy config change.

## Finding 5: C4's own risk analysis chases the wrong client — mobile never calls `/api/ics`, but web users losing their cookie (a more likely case) isn't analyzed at all
- **Severity:** Medium
- **Location:** Phase 4, section "Architecture" step 2 and "Risk Assessment"
- **Flaw:** The plan spends its entire Risk Assessment and a mandatory pre-implementation grep step worrying whether "app native" (`mobile/`) calls `/api/ics/<code>` without a session, potentially breaking .ics downloads on mobile. Actual grep of `mobile/src` shows `mobile/src/calendar.js` never calls the HTTP `/api/ics` endpoint at all — it writes events directly to the device calendar via `expo-calendar` (`Calendar.createEventAsync`), using appointment data already returned from `/api/chat`/`/api/start`. So the flagged "rủi ro lớn nhất" (phase-04 lines 108-111) does not exist. Meanwhile, the plan does not analyze the much more mundane risk for the actual consumer of this endpoint: the web demo's `<a href="/api/ics/{code}">` link rendered inline in the chat (chatbot.py:449,456). Any web user who cleared cookies, opened the link in a different browser/private window, bookmarked it for later, or forwarded it to another device will now get 404 even though they are the legitimate booker — because `resolve_sid()` (app.py:31-38) mints a brand-new random `sid` whenever no cookie/body session is present, which will never match `appt["session"]`.
- **Failure scenario:** A patient books via the web chat, later reopens the .ics link from browser history after their session cookie expired/was cleared (Flask's default session cookie has no explicit `PERMANENT_SESSION_LIFETIME` set in app.py, so behavior depends on browser session-cookie handling) — gets a silent 404 with no explanation, perceived as a broken feature, not a security fix.
- **Evidence:** `mobile/src/calendar.js:1-70` (no HTTP call to `/api/ics`); `chatbot.py:449,456` (ics link rendered client-side in same page/session); `app.py:31-38` (`resolve_sid` always mints fresh `sid` when missing).
- **Suggested fix:** Re-scope the Risk Assessment to the actual consumer (web browser link), and decide/document the intended UX for cookie loss (e.g., re-issue the link with a signed token instead of relying purely on the ambient session cookie) rather than leaving it as an unexamined side effect of the ownership check.

## Findings Summary
1. **Critical** — Phase 2: new partial UNIQUE index can fail at `init_schema()` if duplicate confirmed `(date,time)` rows already exist in prod, taking down the entire app with no migration/cleanup step planned. (`storage.py:91-99`)
2. **Critical** — Phase 1: pattern-side normalization omits `_normalize()` (case folding), so DB-sourced (admin-editable) emergency/diagnosis/handoff patterns with any non-lowercase text silently stop matching — undermines the exact life-safety fix this phase claims to deliver. (`triage.py:40-45`, `storage.py:301-313`)
3. **High** — Phase 3: `reset_session()` doesn't call `move_to_end()`, decoupling OrderedDict eviction order from the `_last_seen` recency field, so actively-reset sessions can be evicted ahead of genuinely idle ones. (`chatbot.py:41-42,63,84`)
4. **Medium** — Plan-wide: "1 Flask process" assumption behind the whole C3 design has zero enforcement or documentation in the repo (no Procfile/gunicorn config found); a future multi-worker deploy silently multiplies the memory ceiling with no warning.
5. **Medium** — Phase 4: Risk Assessment investigates a mobile-app risk that grep proves doesn't exist (mobile never calls `/api/ics`), while missing the real, more probable web-cookie-loss 404 regression for the endpoint's actual consumer (`chatbot.py:449,456`, `app.py:31-38`).
