## Finding 1: UniqueViolation handler conflates two distinct constraints, can crash with a leaked traceback
- **Severity:** Critical
- **Location:** Phase 2, section "Architecture" step 3 / "Implementation Steps" step 2
- **Flaw:** `appointments.code` is already `TEXT PRIMARY KEY` (storage.py:46). Phase 2 adds a second UNIQUE index on `(date, time) WHERE status='confirmed'` (storage.py SCHEMA_SQL, proposed at storage.py:44-84 region). The proposed `booking.book_appointment` fix catches `psycopg.errors.UniqueViolation` generically and assumes it always means "slot race lost", then calls `_confirmed_at(date_str, time_str)` to fetch the winning row and build the existing "already booked" response.
- **Failure scenario:** `_generate_code()` (booking.py:66-67) draws from a 36^6 ≈ 2.2B space with no collision-retry loop. A `code` collision — while rare — raises the SAME exception class (`UniqueViolation`, on `appointments_pkey`, not `ux_appointments_slot`). The handler doesn't distinguish which constraint fired: it calls `_confirmed_at(date_str, time_str)` anyway, which returns `None` (the date/time slot genuinely isn't taken), and then the plan's own described code does `taken.get("patient_phone")` → `AttributeError: 'NoneType' object has no attribute 'get'`, raised *while already handling* the UniqueViolation. This new exception is not caught anywhere and propagates to the Flask route as an unhandled 500. `app.py:160` runs `app.run(debug=True, ...)` unconditionally (hardcoded, not env-gated — see ISSUES.md H7), so Werkzeug's interactive debugger/traceback would be returned to the caller, potentially exposing internals.
- **Evidence:** storage.py:46 (`code TEXT PRIMARY KEY`); booking.py:66-67 (`random.choices`, no retry-on-collision); phase-02 plan text: "gọi lại `_confirmed_at(date_str, time_str)` để lấy bản ghi vừa thắng race" with no branch for "constraint wasn't the slot index"; app.py:160 (`debug=True`).
- **Suggested fix:** Inspect `exc.diag.constraint_name` (psycopg exposes this) and only treat `ux_appointments_slot` violations as "slot taken"; re-raise (or retry code generation) for `appointments_pkey` violations. Add a test that simulates a PK-collision `UniqueViolation` distinctly from a slot-collision one — the current test plan (`test_book_appointment_catches_integrity_error`) only simulates the slot case, so it would pass despite this defect (phantom-test risk).

## Finding 2: Session cap/eviction is not atomic — TOCTOU race under any multi-threaded deployment, and the plan's own "1 process" premise doesn't rule this out
- **Severity:** High
- **Location:** Phase 3, section "Architecture" (`get_session()` steps 1-3)
- **Flaw:** `get_session()` is specified as a multi-statement sequence: check membership/expiry → `move_to_end()` or evict via `len(SESSIONS) >= _MAX_SESSIONS` then `SESSIONS.popitem(last=False)` → `SESSIONS[session_id] = _new_session()`. None of these steps are wrapped in a lock. `collections.OrderedDict` methods are individually atomic under CPython's GIL, but the *sequence* (check-len → decide-evict → popitem → insert) is not — the GIL can be released between bytecode ops (e.g., on `sys.setswitchinterval` boundaries), so two threads handling concurrent requests can interleave.
- **Failure scenario:** If the app is deployed with a threaded WSGI server (e.g., `gunicorn --workers 1 --threads 4`, which still satisfies the plan's stated "1 Flask process" decision), two threads creating new sessions near the cap can both read `len(SESSIONS) >= _MAX_SESSIONS` as `False` before either inserts, causing the dict to exceed `_MAX_SESSIONS` momentarily, or both decide to evict and pop two entries for what should be a single net insert, silently dropping a legitimate, recently-active session that neither thread intended to evict.
- **Evidence:** ISSUES.md:83 lists this exact unresolved question — "Production chạy 1 Flask worker hay nhiều worker (gunicorn)? (ảnh hưởng C3/H9)" — and ISSUES.md:67 separately flags "Sửa session dict không có khóa (lock) khi Flask threaded → state có thể bị ghi đè" as a still-open Medium item. plan.md's "Quyết định đã chốt với user" (lines 24-26) only settles *process count*, never addresses *thread count*, yet Phase 3 is built entirely on the "1 process → no lock needed" premise. chatbot.py:16 confirms `SESSIONS = {}` is process-global shared mutable state with no synchronization primitive anywhere in chatbot.py.
- **Suggested fix:** Either add a `threading.Lock` around the check-evict-insert sequence (cheap, always correct regardless of deployment), or explicitly confirm and document in the plan that the WSGI server will run single-threaded (`--threads 1`) as a hard deployment constraint, and add a smoke check/assertion for it.

## Finding 3: Cap+TTL converts unbounded-memory DoS into a trivial legitimate-session-eviction DoS — not covered by the phase's test plan
- **Severity:** High
- **Location:** Phase 3, section "Overview" and "Success Criteria"
- **Flaw:** `_MAX_SESSIONS = 2000` is a hard global cap shared by every user of the app. `app.resolve_sid()` (app.py:31-38) accepts a client-supplied `session` value or mints one with no authentication, and `/api/start` (app.py:49-55) calls `chatbot.start(sid)` → `get_session`/`reset_session` for whatever `sid` was resolved. Nothing rate-limits distinct new sessions.
- **Failure scenario:** A single attacker script calling `POST /api/start` ~2000+ times in a few seconds (trivial HTTP throughput) will evict every real user's in-progress conversation, since LRU evicts the least-recently-touched entry and every attacker-created session is, by construction, more recently touched than any real user's session. The phase's stated fix goal is "chặn trần cứng bất kể client gửi gì" (memory bound) — which it does achieve — but Overview implicitly frames this as solving the DoS described in C3 ("client có thể tạo vô hạn entry → OOM"); it only prevents the OOM outcome, not the availability-DoS outcome (kicking every legitimate user mid-booking). This gap is not mentioned in Risk Assessment and none of the three specified tests (`test_session_cap_evicts_oldest`, `test_session_ttl_expires`, `test_get_session_refreshes_recency`) exercise adversarial mass-eviction of legitimate sessions.
- **Evidence:** phase-03 plan lines 12-14 (DoS framing), 43-56 (Architecture with no rate limit), app.py:31-38, app.py:49-55; phase-03 "Implementation Steps" section lists only cap/TTL/recency tests, no eviction-under-attack test.
- **Suggested fix:** At minimum, document this residual risk explicitly in the plan/ISSUES.md rather than implying C3 is fully closed; consider rate-limiting `/api/start` or requiring session continuity (e.g., don't create a brand-new entry for every unauthenticated hit) as a follow-up, since the plan itself declares rate limiting out of scope elsewhere (Medium item in ISSUES.md:65).

## Finding 4: Phase 5's broadened exception scope silently converts a loud crash into a repeating duplicate-notification bug
- **Severity:** High
- **Location:** Phase 5, section "Architecture" (proposed `scan_once` code block)
- **Flaw:** The proposed try/except wraps the ENTIRE per-appointment body, including both `_send_for(appt, rule)` (which calls `push.send_push` then `booking.mark_reminder_sent`) and the `n_sent += 1` increment. `_send_for` (reminder_worker.py:59-67) calls `push.send_push(...)` first, then unconditionally `booking.mark_reminder_sent(appt["code"], rule["key"])` second, with no error handling of its own.
- **Failure scenario:** If `push.send_push` succeeds (notification delivered) but `booking.mark_reminder_sent` subsequently raises (e.g., transient DB error, matches the exact H2/H3 dedup-integrity problem already noted in ISSUES.md:38-42 as separately-tracked High items) — under the CURRENT code this exception propagates and crashes `scan_once`/`--watch`, a loud, hard-to-miss failure that forces investigation. Under the PROPOSED fix, this same failure is caught by the new broad `except Exception`, logged as an indistinguishable `[SKIP] ... lỗi xử lý nhắc lịch` line, and swallowed. Because `reminders_sent` was never updated, the SAME reminder will be re-sent on every subsequent `--watch` scan (every 60s) for as long as the underlying `mark_reminder_sent` failure persists — producing a growing stream of duplicate push notifications to the same patient, silently, with only a generic SKIP log line (indistinguishable from genuinely malformed appointment data) to signal it.
- **Evidence:** reminder_worker.py:59-67 (`_send_for` ordering: send then mark), reminder_worker.py:100-106 (`--watch` loop, 60s cadence, no dedup outside `reminders_sent`); phase-05 plan Architecture block places `_send_for(appt, rule)` inside the newly-broadened try; phase-05 Risk Assessment (lines 99-102) only generically acknowledges "may hide real bugs" without identifying this specific repeat-notification-storm mechanism.
- **Suggested fix:** Keep the broad per-appointment catch for the *rule discovery / date parsing* portion (as originally intended for KeyError-on-bad-data), but wrap `_send_for` itself more narrowly, or log at a distinguishable severity (e.g., `[ERROR]` vs `[SKIP]`) when the exception originates after a push was already sent, so on-call staff can distinguish "bad data, skip forever" from "send succeeded but bookkeeping failed, will resend forever."

## Finding 5: Phase 4's ownership check has no test/verification for the case where `resolve_sid` mints a brand-new sid on GET, and the plan hand-waves the mobile risk without checking the actual delivery path
- **Severity:** Medium
- **Location:** Phase 4, section "Architecture" step 2, and Implementation Step 1
- **Flaw:** The plan instructs verifying mobile impact by `grep -rn "api/ics" mobile/src` and treating an empty result as proof the native app is unaffected. This is a weak verification method: the `.ics` link is not hardcoded in mobile source — it is generated server-side in `chatbot.py:449` (`ics_url = f"/api/ics/{payload['code']}"`) and only reaches the client embedded as HTML (`chatbot.py:456`, an `<a href=...>` tag) inside the chat `reply` text. A grep for the literal string `api/ics` in `mobile/src` would correctly return nothing (as verified) purely because the mobile app happens to strip `<a>` tags in `mobile/src/html.js:11` — a fact the plan never actually confirms; it only checks for a string that could never have existed there. If a future refactor changes how mobile renders chat HTML (e.g., adds link rendering), this verification step would give false confidence again in a later, unrelated change, since it doesn't test the actual runtime path.
- **Evidence:** chatbot.py:447-456 (ics link generation, HTML `<a>` embedded server-side); mobile/src/html.js:1-31 (explicitly strips `<a>`/`<span>`/`<i>` tags, comment: "Bỏ qua thẻ link (push lo phần nhắc lịch)"); phase-04 plan Implementation Step 1: "grep -rn 'api/ics' mobile/src ... Kết quả quyết định có cần thêm ?session= query param hỗ trợ hay chỉ cookie là đủ."
- **Suggested fix:** State the actual verified mechanism in the plan/report (mobile strips the `<a>` tag entirely, so the ics feature is web-only today) rather than relying on a string-grep proxy that happens to produce the right answer for the wrong reason; this keeps the finding-completion report accurate for future maintainers who may re-check "why is mobile safe" and only see a grep command, not the real reason.

## Finding 6: Phase 2's own risk section acknowledges no real Postgres verification, yet the plan's central premise ("production always has DATABASE_URL") is the sole justification for skipping the JSON-branch fix
- **Severity:** Medium
- **Location:** Phase 2, "Risk Assessment", and plan.md lines 26-28
- **Flaw:** plan.md states as a closed decision: "Production **luôn có `DATABASE_URL`**... C2 chỉ cần fix nhánh Postgres." Phase 2's own Risk Assessment then admits: "Không có Postgres local để test thật... Ghi rõ trong báo cáo nếu chưa chạy được integration test thật trên Postgres." This means the ENTIRE mechanism this phase is supposed to deliver (a real UNIQUE-index-backed race fix) ships verified only via `SCHEMA_SQL` string-matching and a monkeypatched exception — never against an actual concurrent-write Postgres instance. Combined with Finding 1 (constraint-conflation bug), there is a realistic chance the fix has never actually been exercised against the real failure mode (two concurrent real double-booking requests hitting Postgres) before being marked complete.
- **Evidence:** phase-02 plan lines 97-99 ("Không có Postgres local để test thật: rủi ro chính..."); plan.md lines 26-28 (decision framing); phase-02 lines 60-75 (test strategy is entirely monkeypatch-based unless `DATABASE_URL` happens to be present, in which case it's `pytest.mark.skipif`-guarded, i.e., optional).
- **Suggested fix:** Before marking C2 as verified in ISSUES.md, require an actual run against a real (even ephemeral/local Docker) Postgres instance with two concurrent booking calls, not just unit-level monkeypatch tests — otherwise this is a critical life-of-business fix landing without ever exercising its real code path.

## Full Findings List (summary)

1. **Critical** — Phase 2: `UniqueViolation` handler doesn't distinguish PK collision (`code`) vs slot collision (`date,time`); can crash with `AttributeError` inside the except block, unhandled, leaking a traceback since `app.py:160` runs `debug=True` unconditionally. Not caught by the phase's own test plan.
2. **High** — Phase 3: session cap/eviction (`OrderedDict` check→evict→insert) is not atomic; races under any multi-threaded deployment, a scenario the plan's "1 process" decision never actually rules out (ISSUES.md:83 explicitly leaves worker/thread count as an open question).
3. **High** — Phase 3: hard global session cap with no rate limiting converts an OOM-DoS into a trivial mass-eviction-of-legitimate-sessions DoS; not tested or acknowledged as a residual risk.
4. **High** — Phase 5: broadening the try/except to wrap `_send_for` silently converts a loud crash-on-dedup-failure into a repeating duplicate-push-notification bug, indistinguishable in logs from benign bad-data skips.
5. **Medium** — Phase 4: mobile-impact verification step (`grep mobile/src` for `api/ics`) tests for the wrong signal; it happens to give the right answer only because mobile strips `<a>` tags elsewhere, a fact never actually confirmed in the plan.
6. **Medium** — Phase 2: the entire Postgres-only fix strategy is justified by an unverifiable production assumption and is planned to ship without ever running against a real concurrent-write Postgres instance.
