# Red Team Review: consolidate-docs plan (Failure Mode Analyst)

## Finding 1: Shell-command paths inside moved tutorial files are never fixed (hoc/00, hoc/01)
- **Severity:** Critical
- **Location:** Phase 1, section "Bước 3 — sửa link nội bộ trong 6 bài hoc/*.md" (and plan.md's "Link nội bộ" section)
- **Flaw:** Phase 1's link-fix scope is limited to 6 files' `[label](../app/x.py)` markdown links plus one `[DATABASE.md]` link in `hoc/08`. It never touches `hoc/00-muc-luc.md` or `hoc/01-viet-triage-tu-dau.md`, which contain literal shell commands and prose telling the reader to create and run `hoc/triage_demo.py` from the repo root (established via `cd /Users/hieutm3/Desktop/PRF-SHI` at line 15 of hoc/01).
- **Failure scenario:** After Phase 1 runs `git mv hoc/*.md docs/hoc/` and removes the empty `hoc/` directory, a student following `docs/hoc/01-viet-triage-tu-dau.md` verbatim will run `touch hoc/triage_demo.py` and `./.venv/bin/python hoc/triage_demo.py` from repo root. `hoc/` no longer exists at root, so `touch` silently recreates a stray root-level `hoc/` directory (undoing the exact cleanup this plan performs) or the python invocation fails with "No such file or directory" — breaking the tutorial's worked example.
- **Evidence:** `hoc/01-viet-triage-tu-dau.md:19,21,24,31,48,238` — 6 occurrences of `hoc/triage_demo.py` as literal shell/file paths, none listed in Phase 1's "Related Code Files" (`plans/260710-1200-consolidate-docs/phase-01-move-docs-and-fix-internal-links.md:95-104`, which only lists the app/*.py + DATABASE.md link fixes). `hoc/00-muc-luc.md:38,42` has the same pattern (`./.venv/bin/python hoc/<file_ban_tao>.py`, `hoc/triage_demo.py`). Confirmed via `grep -n "hoc/" hoc/00-muc-luc.md hoc/01-viet-triage-tu-dau.md`.
- **Suggested fix:** Add a Phase 1 step (or expand Phase 3) to grep all `hoc/*.md` (post-move, in `docs/hoc/`) for literal `hoc/` path references in shell-command code blocks and prose, and decide/document the new convention (e.g. `docs/hoc/triage_demo.py`, or relocate practice-file guidance out of `docs/` entirely since `docs/` is meant for documentation, not scratch code).

## Finding 2: Plan conflates markdown-link relative resolution with shell-command CWD-relative resolution
- **Severity:** High
- **Location:** Phase 3, section "docs/getting-started-guide.md" Architecture + Implementation Step 4
- **Flaw:** Phase 3's classification heuristic ("bare `hoc/...md` in a file already under `docs/` is coincidentally correct after the move") is valid ONLY for markdown-link paths, which resolve relative to the containing `.md` file's directory. It is invalid for shell-command paths, which resolve relative to the terminal's CWD (repo root, per `docs/getting-started-guide.md:52` `cd /Users/hieutm3/Desktop/PRF-SHI`). The plan's own grep (`grep -n "hoc/\|..." docs/getting-started-guide.md`) will surface line 285 (`./.venv/bin/python hoc/triage_demo.py`) alongside real markdown links, but the plan's stated classification rule doesn't distinguish "shell command" from "markdown link" as a category at all.
- **Failure scenario:** An implementer mechanically applying Phase 3's stated rule ("Nếu đang là hoc/01-....md (bare, không ../) → ... tình cờ ĐÚNG. Không sửa") to line 285's `hoc/triage_demo.py` will leave it unchanged, believing it is "coincidentally correct" like the markdown links. It is not — the command executes from repo root, not from `docs/`, so it needs to become `docs/hoc/triage_demo.py` (or be redesigned). The plan flags line 395 as a special "1 câu" (singular) prose case needing a prefix edit, undercounting: there are two occurrences with different resolution semantics (line 285 command, line 395 prose), and neither the Architecture section's classification table nor the implementation steps name "shell command" as a distinct category requiring different resolution logic than markdown links.
- **Evidence:** `docs/getting-started-guide.md:52` (`cd /Users/hieutm3/Desktop/PRF-SHI`), `:285` (`./.venv/bin/python hoc/triage_demo.py`), `:395` (prose mention); Phase 3 file `phase-03-update-getting-started-and-roadmap-docs.md:41-58` only discusses markdown-link depth and treats the triage_demo.py mention as a single generic "example," not a CWD-relative runnable command.
- **Suggested fix:** Explicitly split Phase 3's classification into "markdown link (resolve relative to containing file)" vs "shell command / code path (resolve relative to CWD after the last `cd`)" and audit both categories separately for every `hoc/` occurrence in `docs/getting-started-guide.md`.

## Finding 3: `evaluate.py` re-run for TDD verification will overwrite non-deterministic content, but the plan's own acceptance criterion forbids content changes
- **Severity:** High
- **Location:** Phase 1, "Implementation Steps (TDD)" steps 1 and 6; plan.md Acceptance Criteria (lines 127-128)
- **Flaw:** The Red/Green verification mandates running `python3.10 eval/evaluate.py` before AND after the move. `eval/results.md` contains a "Thời gian TB (ms/câu)" (average per-query latency) column that is freshly computed on every run and is inherently non-deterministic (depends on machine load). Re-running the script per the TDD steps will therefore write a `docs/eval/results.md` whose content differs from the git-mv'd baseline — not merely relocated, but changed — directly contradicting the plan's own top-level acceptance criterion "KHÔNG viết lại nội dung bất kỳ file `.md` nào ... giữ nguyên 100% nội dung khác."
- **Failure scenario:** Implementer follows the TDD steps exactly, runs `evaluate.py` in the Green step to "verify" the `RESULTS_PATH` fix, and the resulting `docs/eval/results.md` has different timing numbers than the version that was `git mv`'d — producing an unintended content diff in what was supposed to be a pure path/rename commit, violating the plan's explicit non-functional requirement.
- **Evidence:** `eval/results.md:11` (grep-confirmed content includes a `Thời gian TB (ms/câu)` row with per-run-computed values); plan.md:127-128 acceptance criterion; Phase 1 steps 1 and 6 (`phase-01-move-docs-and-fix-internal-links.md:108-123`) mandate re-running `evaluate.py` twice as the verification method.
- **Suggested fix:** Either explicitly carve out an exception in the acceptance criteria for the regenerated timing metric, or verify the `RESULTS_PATH` fix with a lighter-weight method (e.g. asserting the write-target path directly, without requiring a full re-run that mutates tracked content) that doesn't overwrite the already-moved report.

## Finding 4: Phase 2/Phase 3 parallel execution has no explicit gate confirming Phase 1's `git mv` sequence is fully committed before either starts
- **Severity:** Medium
- **Location:** plan.md, "Phases" table and "Phase 1 PHẢI xong trước — Phase 2 và 3 chạy song song sau đó"
- **Flaw:** Phase 2 and Phase 3 are declared safe to run in parallel because they touch disjoint files. That's true for write conflicts, but both phases' correctness depends on reading the actual *post-move* filesystem state to classify links as "coincidentally correct" vs. "needs `../` removed." Both phase files rely on `test -f`-style checks against the current tree. If Phase 1's `git mv` sequence (5 separate `git mv`/`git rm` invocations plus 2 file edits) is only partially applied when a Phase 2/3 agent starts — e.g. `hoc/` moved but `BAOCAO_DANHGIA.md` not yet moved — link classification will silently use a mixed old/new tree state and produce wrong "already correct, don't touch" conclusions.
- **Failure scenario:** A Phase 3 agent checks `test -f ../BAOCAO_DANHGIA.md` from `docs/project-roadmap.md`'s perspective mid-Phase-1-execution; the old-location file still exists (git mv not yet run for that specific file), so the check misleadingly "passes," and the agent leaves the `../BAOCAO_DANHGIA.md` link untouched — only for Phase 1 to complete moments later and break that link, with Phase 3 already marked done.
- **Evidence:** plan.md:88-93 lists Phase 2/3 dependency as "Phase 1" via frontmatter `dependencies: [1]` in both phase files, but neither phase file nor plan.md specifies a hard synchronization checkpoint (e.g., "confirm Phase 1's Success Criteria checklist is fully checked and `git status` is clean" before launching Phase 2/3).
- **Suggested fix:** Add an explicit gate: Phase 2/3 must not start until Phase 1's Success Criteria are all checked off and `git status` confirms no pending/partial renames — not merely "Phase 1 marked done" in a task tracker.

## Finding 5: Verification method proves link targets exist, not that the edited files actually contain the intended (correct) paths
- **Severity:** Medium
- **Location:** Phase 1 Implementation Step 6 (verification loop); plan.md Acceptance Criteria
- **Flaw:** All three phases' verification strategy is `test -f <resolved-path>` against a path list written directly into the phase file (the "expected correct" paths), not derived by extracting the actual link targets from the edited markdown files after editing. This proves the *intended* target exists on disk; it does not prove the edited file's actual link text matches that intended target. A typo introduced during manual editing (e.g. `../../app/tirage.py` instead of `../../app/triage.py`) would go undetected because the verification loop tests the plan's hardcoded correct-path list, not what ended up written in the file.
- **Failure scenario:** Implementer edits `docs/hoc/01-viet-triage-tu-dau.md` and mistypes the replacement link target. The Phase 1 verification loop (`for f in ../../app/triage.py ...; do test -f "$f" ...`) still reports "OK" for every entry because it's checking a fixed list of correct paths copied from the plan text, completely independent of what the edited file now contains. The success criteria checklist gets checked off despite the actual link being broken.
- **Evidence:** `phase-01-move-docs-and-fix-internal-links.md:124-131` — the verification loop iterates a literal hardcoded list (`../../app/triage.py ../../app/data.py ...`), not values extracted via `grep -oP '\(\K[^)]+(?=\))'` from the actual edited `docs/hoc/*.md` files.
- **Suggested fix:** Change verification to extract actual link targets from the edited files programmatically (e.g. `grep -oP '\]\(\K[^)]+' docs/hoc/01-*.md`) and test those extracted paths, rather than testing a pre-written "expected" path list that can pass even when the file's real content diverges from it.

## Finding 6: `git mv` for `eval/results.md` assumes the auto-generated results file is tracked by git; no pre-check for `.gitignore` exclusion
- **Severity:** Medium
- **Location:** Phase 1, Architecture "Bước 1 — git mv"; Risk Assessment bullet 2
- **Flaw:** plan.md describes `eval/results.md` as a file "DO eval/evaluate.py TỰ SINH" (auto-generated by evaluate.py on every run). Auto-generated evaluation/report artifacts are a common candidate for `.gitignore` exclusion in Python projects. `git mv` only operates on tracked files — if `eval/results.md` is untracked, the `git mv eval/results.md docs/eval/results.md` command in Phase 1's single bash block will error ("not under version control"), and depending on how the block is executed (one shell invocation vs. sequential separate commands), this could abort the remaining `git mv`/`git rm` commands in the same block or leave a silently inconsistent partial move.
- **Failure scenario:** If `eval/results.md` turns out to be gitignored, the Bước 1 bash block's `git mv eval/results.md docs/eval/results.md` line fails mid-sequence; if run as one shell script without `set -e` or without per-line error checking, subsequent commands (`git mv BAOCAO_DOAN.md ...`, `git rm release-manifest.json ...`) may still execute, leaving a partially-completed move that the plan's "git status shows renamed:" risk check would misinterpret (no rename entry for results.md is expected in that check, but for the wrong reason — untracked file, not successful move).
- **Evidence:** plan.md:43 ("results.md ... file DO eval/evaluate.py TỰ SINH"); Phase 1 Architecture Bước 1 (`phase-01-move-docs-and-fix-internal-links.md:28-38`) presents all `git mv` commands as one bash block with no tracked-file pre-check and no per-command failure handling specified.
- **Suggested fix:** Add a pre-check step, `git ls-files eval/results.md` (or inspect `.gitignore` for `results.md`/`eval/*.md` patterns), to confirm the file is tracked before attempting `git mv`; if untracked, fall back to plain `mv` + `git add` instead, and treat the bash block as a sequence with per-command failure verification rather than an atomic assumption.
