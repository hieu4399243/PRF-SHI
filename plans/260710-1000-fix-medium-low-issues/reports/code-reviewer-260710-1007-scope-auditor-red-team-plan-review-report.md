# Red Team Review — Scope Auditor Role (State Lifetime)

Plan: `plans/260710-1000-fix-medium-low-issues/`

## Finding 1: Per-session `_lock` is bound to a dict instance that `/reset` silently replaces, defeating M10's serialization guarantee
- **Severity:** Critical
- **Location:** Phase 1, section "M10 — khoá per-session cho field-level write"
- **Flaw:** The plan proposes `with sess["_lock"]:` wrapped around the *entire* `handle_message()` body, captured once at function entry from the `sess` object returned by `get_session()`. But `handle_message()` already contains a branch (the `/reset` command) that calls `reset_session(session_id)` mid-function, which deletes the old dict from `SESSIONS` and inserts a brand-new dict (`_new_session()`) — with a brand-new, unlocked `threading.Lock()` instance — under the same `session_id`.
- **Failure scenario:** Thread A calls `handle_message(sid, "/reset")`. It fetches `sess=D1` (lock `L1`), enters `with L1:`. Inside the `/reset` branch (`chatbot.py:127-134`) it calls `reset_session(sid)`, which swaps `SESSIONS[sid]` to a new dict `D2` (lock `L2`, unlocked), then reassigns the local variable `sess = get_session(session_id)` (line 129) to `D2` and continues writing `sess["state"] = resp["state"]` (line 132) — all while the `with` context manager is still holding `L1`, not `L2`. Meanwhile Thread B calls `handle_message(sid, msg)` for the *same* `session_id`, fetches `D2` via `get_session()`, acquires `L2` (unlocked, succeeds immediately), and starts mutating `D2`'s fields concurrently with Thread A's post-reset writes to the same `D2` dict. This is exactly the double-submit interleaving M10 is supposed to prevent, and it happens on a code path that already exists today.
- **Evidence:** `chatbot.py:78-83` (`reset_session` creates a fresh dict via `_new_session()` and reassigns `SESSIONS[session_id]`), `chatbot.py:127-134` (`/reset` branch calls `reset_session` then `get_session` mid-`handle_message`, reassigning local `sess`). The same dict-swap-under-lock pattern also exists in the TTL-expiry path at `chatbot.py:70-74` inside `get_session()` itself.
- **Suggested fix:** Do not rely on a lock object stored inside a replaceable dict. Use a lock keyed by `session_id` in a separate, stable structure (e.g., a `dict[str, threading.Lock]` protected by `_SESSIONS_LOCK` for its own creation/eviction), or make `reset_session()` reuse the *existing* lock object instead of creating a new one when a session with that id already exists.

## Finding 2: `M2`'s "every JSON read-modify-write wrapped in `_JSON_LOCK`" claim is false — `add_token` is left unlocked while the new `remove_token` (same file) is locked
- **Severity:** High
- **Location:** Phase 2, section "M2 — atomic write + lock cho nhánh JSON" and "M5 — parse ticket Expo + xoá token hết hạn"
- **Flaw:** The plan's Related Code Files list for `storage.py` only names `_json_save`, `add_appointment`, `set_reminder_sent`, `set_status`, plus the new `DuplicateCodeError`/`remove_token`. It never lists `add_token` as modified, yet `add_token` performs the identical unlocked read-modify-write on `TOKENS_PATH` (`device_tokens.json`) that the new `remove_token` will now protect with `_JSON_LOCK`.
- **Failure scenario:** `push.send_push()` (Phase 2, M5) calls `storage.remove_token(token)` under `_JSON_LOCK` when Expo reports `DeviceNotRegistered`. Concurrently, `/api/register-push` → `push.register_token()` → `storage.add_token()` (`storage.py:289-306`, no lock) does its own unlocked `_json_load` → mutate → `_json_save` on the same `TOKENS_PATH`. If `add_token`'s unlocked read happens before `remove_token`'s locked write, and its unlocked write happens after, `add_token` silently resurrects/overwrites the just-removed token (lost update) — the exact class of bug M2 claims to close for "every" JSON read-modify-write function, contradicted by the plan's own Success Criteria line "`mọi thao tác JSON đọc-sửa-ghi bọc _JSON_LOCK`".
- **Evidence:** `storage.py:289-306` (`add_token`, unlocked `_json_load`/`_json_save` on `TOKENS_PATH`), `push.py:34-36` (`register_token` calls `storage.add_token`), Phase 2 plan `Related Code Files` section (omits `add_token`).
- **Suggested fix:** Add `add_token` to the Related Code Files / Architecture list and wrap it in `_JSON_LOCK` in the same pass, since it shares a JSON file with the newly-locked `remove_token`.

## Finding 3: M6 architecture pseudocode does not match the actual `_insert_with_race_guard` except-structure it claims to only "insert into"
- **Severity:** Medium
- **Location:** Phase 2, section "M6 — phát hiện trùng code ở JSON mode"
- **Flaw:** The plan shows a two-clause structure (`except storage.DuplicateCodeError: ... except Exception as exc: ... else: return True, appointment`), but the real function has a *single* `except Exception as exc:` block (`booking.py:180-206`) that internally does a nested `import psycopg` + `isinstance(exc, psycopg.errors.UniqueViolation)` check, and has no `else:` clause — the success path is just the code after the `try/except`, not an `else` branch. Implementation Steps says "Đọc kỹ cấu trúc try/except hiện tại... không viết lại từ đầu" but the shown snippet is not achievable without restructuring the existing block into multiple `except` clauses.
- **Failure scenario:** An implementer pattern-matching the snippet literally would need to significantly rewrite (not just insert into) `_insert_with_race_guard`, risking loss of the existing nested psycopg-import-guard / constraint-name branching logic (`booking.py:183-197`) that the plan itself says must stay untouched.
- **Evidence:** `booking.py:180-206` (actual current structure — one `except Exception as exc:` block, nested isinstance check, no `else`) vs. Phase 2 plan's Architecture snippet showing two separate `except` clauses and an `else:`.
- **Suggested fix:** Rewrite the architecture snippet to add `except storage.DuplicateCodeError:` as a new clause inserted before the existing single `except Exception as exc:` block, without altering the internal psycopg/constraint-name logic, and show the retry logic reusing the existing recursive call pattern verbatim.

## Finding 4: New M1 diagnosis guardrail is inserted before existing cancel/info-question branches, silently overriding them for CONFIRM_DEPT/DONE states
- **Severity:** Medium
- **Location:** Phase 1, section "M1 — guardrail chẩn đoán áp dụng mọi state trừ TRIAGE"
- **Flaw:** The plan places the new guardrail "ngay sau khối `needs_human_handoff`... TRƯỚC phần định tuyến theo state" (i.e., before line 173). But `handle_message()` already has cancel-intent detection (`chatbot.py:155-160`) and info-question detection (`chatbot.py:164-171`) positioned there, both of which apply to `TRIAGE, CONFIRM_DEPT, DONE`. The new guardrail's condition (`sess["state"] != "TRIAGE"`) still covers `CONFIRM_DEPT` and `DONE`, so it will fire for those states too — ahead of the cancel/info-question checks.
- **Failure scenario:** A user in `CONFIRM_DEPT` or `DONE` state asks an info-question or cancel-intent message that also happens to match `safety.is_diagnosis_request()` (rule-based, so false positives are acknowledged by the plan itself in its own Risk Assessment). Previously this reached `triage.info_question_service()` / `_is_cancel_request()`; after this change it will be intercepted by the new guardrail instead, silently changing existing non-TRIAGE behavior — which contradicts the plan's own non-functional requirement "Không đổi cấu trúc dict trả về của bất kỳ response nào" / implicit no-regression goal, and is not covered by any test in Implementation Steps (tests only check `PICK_TIME` and `TRIAGE`, not `CONFIRM_DEPT`/`DONE` overlap with cancel/info-question).
- **Evidence:** `chatbot.py:155-171` (cancel/info-question blocks apply to `CONFIRM_DEPT`, `DONE`), Phase 1 plan Architecture section (guardrail placed before these blocks, condition is `!= TRIAGE` not `== TRIAGE-adjacent-only`).
- **Suggested fix:** Place the guardrail after the cancel/info-question blocks (right before "Định tuyến theo trạng thái"), or narrow its exclusion set to all states that already have their own free-text handling (`TRIAGE, CONFIRM_DEPT, DONE`), and add a test exercising the overlap case explicitly.

## Finding 5: `_RATE_BUCKETS` global lives for full process lifetime and is shared across unrelated concurrent test suites, but only Phase 3's own test file is instructed to reset it
- **Severity:** Medium
- **Location:** Phase 3, section "M8 — rate limit theo IP" and Risk Assessment
- **Flaw:** `_RATE_BUCKETS` is a module-level global in `app.py`, populated by `@app.before_request` on every request to `/api/start`, `/api/chat`, `/api/register-push`, and `/api/ics/<code>` — including requests made by `tests/test_app_ics.py` and `tests/test_app_admin.py` (existing files from prior plans, not modified by this plan) when the full suite runs via `pytest tests/`. The plan's own Risk Assessment acknowledges this ("cân nhắc liệu test file KHÁC... có bị rate-limit dội ngược") but defers the actual fix to "nếu phát hiện vấn đề này khi implement" rather than mandating it up front, and the Related Code Files for Phase 3 list only `tests/test_app_hardening.py` as touched — no shared conftest-level reset fixture is planned.
- **Failure scenario:** `tests/test_app_ics.py` makes >30 requests to `/api/ics/<code>` across its test functions in a single pytest process; combined with prior test files hitting `/api/start`/`/api/chat` from the same default `remote_addr` (Flask test client uses a fixed loopback address), the shared `_RATE_BUCKETS` counter accumulates across files and can trip `429` mid-suite for tests that have nothing to do with rate limiting, causing spurious failures order-dependent on `pytest tests/` collection order.
- **Evidence:** Phase 3 plan `Risk Assessment` (self-acknowledged, deferred as conditional "nếu phát hiện"), `_RATE_LIMITED_PATHS` includes `/api/ics/` prefix which `test_app_ics.py` exercises repeatedly (file exists per `ls tests/`), Success Criteria only requires `tests/test_app_ics.py`, `tests/test_app_admin.py` "vẫn pass" without mandating the isolation mechanism that would guarantee it.
- **Suggested fix:** Make the `_RATE_BUCKETS.clear()` autouse fixture a hard requirement (not conditional), placed in a shared `conftest.py` (or at minimum documented as mandatory in Implementation Steps) so it resets before *every* test file in the suite, not just `test_app_hardening.py`.

## Finding 6: TTL-expiry path in `get_session()` has the identical dict-swap-under-lock defect as Finding 1, but is untested
- **Severity:** Medium
- **Location:** Phase 1, section "M10", `get_session()` at `chatbot.py:62-75`
- **Flaw:** When a session's `_last_seen` exceeds `_SESSION_TTL_SECONDS`, `get_session()` deletes the old dict and creates a fresh one (`chatbot.py:70-74`), producing a new unlocked `_lock` for the same `session_id` — same root cause as Finding 1's `/reset` case. None of the Phase 1 test list (`test_new_session_has_lock`, `test_concurrent_handle_message_same_session_serialized`, `test_different_sessions_not_blocked_by_each_other`) exercises the TTL-expiry-during-concurrent-access scenario.
- **Failure scenario:** Two near-simultaneous requests arrive for a session_id whose TTL just expired: one thread's `get_session()` call observes expiry and starts a fresh dict/lock while a second thread is (or was, moments earlier) still holding the old lock — the new lock provides no serialization against the tail of the old request's writes to the newly-swapped dict, identical mechanism to Finding 1.
- **Evidence:** `chatbot.py:65-74`, same pattern as `reset_session` at `chatbot.py:78-83`.
- **Suggested fix:** Same as Finding 1 — decouple lock lifetime from dict-instance lifetime.
